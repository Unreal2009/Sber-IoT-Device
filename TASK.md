# Тестовое задание: ESP32-S3 ADC Wi-Fi OTA

## Краткое описание

Необходимо реализовать прошивку для ESP32-S3 на FreeRTOS.

Устройство должно снимать сигнал с генератора через ADC, сохранять значения в RAM, формировать CSV и отправлять его на заданный сервер по Wi-Fi.

Дополнительная часть задания - реализовать OTA-обновление прошивки с сервера.

## Оборудование

Используется плата:

```text
ESP32-S3-DevKitC-1 N16R8
16 MB Flash
8 MB PSRAM
USB
Wi-Fi
BLE
```

К плате подключены:

* RGB-светодиод;
* кнопка;
* генератор сигнала на вход ADC.

SSID и password Wi-Fi можно задать в коде.

## Серверная часть

Для задания используется Java-сервер.

Сервер должен поддерживать:

* прием CSV-файла с измерениями;
* просмотр загруженных CSV-файлов;
* выдачу информации о доступной версии прошивки;
* выдачу бинарного файла прошивки для OTA-обновления.

## Основной сценарий работы

1. После включения устройство запускается, выполняет инициализацию и выводит в консоль текущую версию ПО в формате `v1.0.0`.
2. Во время включения и инициализации светодиод мигает зеленым.
3. После завершения инициализации устройство переходит в режим ожидания.
4. В режиме ожидания светодиод горит зеленым постоянно.
5. После однократного нажатия на кнопку устройство переходит в режим снятия показаний.
6. Во время снятия показаний светодиод мигает синим.
7. Устройство считывает значения ADC и сохраняет их в буфер RAM.
8. Измерение завершается:

   * по повторному нажатию кнопки;
   * или при заполнении буфера заданной длины.
9. После завершения измерения устройство формирует CSV.
10. Устройство подключается к Wi-Fi.
11. Во время подключения к Wi-Fi светодиод горит синим постоянно.
12. После подключения устройство отправляет CSV на заданный URL.
13. Во время передачи CSV светодиод мигает желтым.
14. После успешной отправки устройство возвращается в режим ожидания, светодиод снова горит зеленым постоянно.
15. При ошибке устройство переходит в режим ошибки, светодиод мигает красным пока не нажать кнопку один раз, а причина ошибки выводится в консоль (один раз).

## Индикация состояний

| Состояние                 | Индикация          |
| ------------------------- | ------------------ |
| Включение и инициализация | зеленый мигающий   |
| Ожидание                  | зеленый постоянный |
| Снятие показаний          | синий мигающий     |
| Подключение к Wi-Fi       | синий постоянный   |
| Передача CSV              | желтый мигающий    |
| OTA-обновление            | желтый постоянный  |
| Ошибка                    | красный мигающий   |

Цвета используются как простая индикация состояния устройства:

* зеленый - устройство включено и готово к работе;
* синий - устройство выполняет измерение или подключается к сети;
* желтый - устройство передает данные или выполняет обновление;
* красный - возникла ошибка.

Ошибки должны дополнительно выводиться в консоль.

Примеры:

```text
ERROR: Wi-Fi connection failed
ERROR: CSV upload failed
ERROR: OTA update failed
ERROR: ADC buffer overflow
```

## Консольный вывод версии ПО

При каждом старте устройство должно выводить в консоль текущую версию ПО.

Пример:

```text
Firmware version: v1.0.0
```

После успешного OTA-обновления и перезагрузки строка версии должна измениться.

Пример:

```text
Firmware version: v1.0.1
```

## Этап 1. Старт устройства и вывод версии ПО

Реализовать минимальный запуск прошивки.

Что нужно сделать:

1. Создать проект под ESP32-S3.
2. Добавить константу версии ПО.
3. При старте вывести версию ПО в консоль.
4. Реализовать базовую инициализацию приложения.
5. После инициализации перевести устройство в режим ожидания.

Идеальный результат:

* прошивка собирается и запускается на плате;
* в консоль выводится строка `Firmware version: v1.0.0`;
* устройство явно проходит стадию инициализации;
* после инициализации устройство переходит в стабильное состояние ожидания.

## Этап 2. Обработка кнопки

Добавить обработку кнопки.

Что нужно сделать:

1. Реализовать определение нажатия и отпускания кнопки.
2. Реализовать защиту от дребезга.
3. Определять длительность удержания кнопки (больше 3х секунд).
4. Разделить короткое и длинное нажатие.
5. Основные действия выполнять только после отпускания кнопки.
6. Выводить итоговое событие кнопки в консоль.

Кнопка должна формировать два логических события:

```text
Button short press
Button long press
```

Логика обработки:

* `Button short press` - запуск или остановка измерения;
* `Button long press` - вход в режим OTA-обновления.

Событие считается подтвержденным только после отпускания кнопки. Это нужно, чтобы не запускать действие до того, как стало понятно, короткое это нажатие или длинное.

Пример логов:

```text
Button short press
Button long press
```

Идеальный результат:

* короткое нажатие определяется стабильно;
* длинное нажатие определяется отдельно;
* дребезг не приводит к ложным срабатываниям;
* действие выполняется только после отпускания кнопки;
* короткое действие не срабатывает после длинного нажатия;
* в консоль выводится итоговое событие кнопки;
* код обработки кнопки можно использовать на следующих этапах.

## Этап 3. Режимы светодиода и состояние устройства

Добавить конечный автомат состояний и индикацию.

Что нужно сделать:

1. Реализовать состояния устройства.
2. Связать состояния с режимами светодиода.
3. Переключать состояния по событию кнопки.
4. Реализовать состояние `ERROR`.
5. Обеспечить вывод причины ошибки в консоль один раз при переходе в `ERROR`.
6. Оставлять устройство в состоянии `ERROR` до короткого нажатия кнопки.
7. После короткого нажатия в состоянии `ERROR` сбрасывать ошибку и возвращать устройство в `IDLE`.
8. Выводить смену состояния в консоль.

Рекомендуемые состояния:

```text
INIT
IDLE
MEASURING
WIFI_CONNECTING
UPLOADING
OTA_CHECKING
OTA_UPDATING
ERROR
```

Логика переходов:

* после старта устройство переходит в `INIT`;
* после завершения инициализации устройство переходит в `IDLE`;
* `Button short click` в состоянии `IDLE` переводит устройство в `MEASURING`;
* `Button short click` в состоянии `MEASURING` завершает измерение;
* после завершения измерения устройство переходит к подключению Wi-Fi и отправке CSV;
* если Wi-Fi не подключился, устройство переходит в `ERROR`;
* если CSV не отправился, устройство переходит в `ERROR`;
* `Button long click` запускает OTA-сценарий;
* если OTA-сценарий завершился ошибкой, устройство переходит в `ERROR`;
* `Button short click` в состоянии `ERROR` сбрасывает ошибку и возвращает устройство в `IDLE`.

Индикация состояний:

| Состояние                       | Индикация          |
| ------------------------------- | ------------------ |
| `INIT`                          | зеленый мигающий   |
| `IDLE`                          | зеленый постоянный |
| `MEASURING`                     | синий мигающий     |
| `WIFI_CONNECTING`               | синий постоянный   |
| `UPLOADING`                     | желтый мигающий    |
| `OTA_CHECKING` / `OTA_UPDATING` | желтый постоянный  |
| `ERROR`                         | красный мигающий   |

Состояние `ERROR` не нужно создавать искусственно. Оно должно возникать из реальных сценариев ошибки, например:

* не указаны или неверно указаны Wi-Fi SSID/password;
* сервер недоступен;
* CSV не удалось отправить;
* не удалось получить информацию о версии прошивки;
* не удалось скачать или применить OTA-обновление.

При входе в `ERROR` причина ошибки должна быть выведена в консоль один раз.

Примеры:

```text
ERROR: Wi-Fi connection failed
ERROR: CSV upload failed
ERROR: OTA version check failed
ERROR: OTA update failed
```

После этого устройство остается в состоянии `ERROR`, а светодиод продолжает мигать красным до тех пор, пока пользователь не выполнит короткое нажатие кнопки.

Идеальный результат:

* после старта устройство показывает режим инициализации;
* после инициализации переходит в `IDLE`;
* короткое нажатие переводит устройство из `IDLE` в `MEASURING`;
* повторное короткое нажатие завершает `MEASURING`;
* после завершения измерения устройство переходит к Wi-Fi-подключению и отправке CSV;
* при ошибке Wi-Fi, отправки CSV или OTA устройство переходит в `ERROR`;
* причина ошибки выводится в консоль один раз;
* красная индикация ошибки продолжается до короткого нажатия;
* короткое нажатие в `ERROR` сбрасывает ошибку и возвращает устройство в `IDLE`;
* светодиод всегда отражает текущее состояние устройства.

## Этап 4. Снятие показаний с ADC и формирование CSV

Добавить режим измерения и формирование CSV без перехода к сетевой части.

Что нужно сделать:

1. В режиме `MEASURING` считывать значения ADC.
2. Частота опроса ADC по умолчанию - 1 кГц.
3. Длина буфера по умолчанию - 10000 значений ADC.
4. При частоте 1 кГц буфер соответствует примерно 10 секундам записи.
5. Сохранять значения ADC в буфер RAM.
6. Завершать измерение по повторному короткому нажатию кнопки.
7. Завершать измерение автоматически при заполнении буфера.
8. После завершения измерения сформировать CSV.
9. После формирования CSV вывести в консоль краткий preview результата.

Минимальный формат CSV:

```csv
index,value
0,1234
1,1241
2,1238
```

Для проверки этапа без Wi-Fi и сервера после формирования CSV устройство должно вывести в консоль:

* количество собранных точек;
* размер сформированного CSV в байтах;
* первые несколько строк CSV;
* последние несколько строк CSV.

Полностью печатать CSV в консоль не требуется.

Пример логов:

```text
Measurement started
ADC sample rate: 1000 Hz
ADC buffer size: 10000
ADC samples collected: 10000
Measurement stopped
CSV created
CSV size: 98764 bytes

CSV preview:
index,value
0,1234
1,1241
2,1238
...
9997,1240
9998,1236
9999,1239
```

Идеальный результат:

* данные ADC регулярно считываются;
* частота опроса ADC задается явно;
* размер буфера задается явно;
* буфер не переполняется;
* измерение корректно завершается по кнопке;
* измерение корректно завершается при заполнении буфера;
* CSV формируется из реально снятых данных;
* в консоль выводится количество собранных точек;
* в консоль выводится размер CSV;
* в консоль выводится preview CSV: заголовок, первые строки и последние строки;
* этап можно проверить без подключения к Wi-Fi и без отправки CSV на сервер.

## Этап 5. Подключение к Wi-Fi и отправка CSV

Добавить сетевую часть.

Что нужно сделать:

1. После формирования CSV подключиться к Wi-Fi.
2. SSID и password можно задать в коде.
3. Во время подключения использовать состояние `WIFI_CONNECTING`.
4. Во время подключения светодиод должен гореть синим постоянно.
5. После подключения отправить CSV на заданный URL.
6. Во время передачи использовать состояние `UPLOADING`.
7. Во время передачи светодиод должен мигать желтым.
8. После успешной отправки вернуться в `IDLE`.
9. При ошибке перейти в `ERROR`.

Идеальный результат:

* устройство подключается к Wi-Fi;
* состояние подключения видно по светодиоду;
* передача CSV отличается по индикации от подключения;
* CSV успешно приходит на Java-сервер;
* после отправки устройство возвращается в режим ожидания;
* ошибки подключения и отправки выводятся в консоль;
* при ошибке светодиод мигает красным.

Пример логов:

```text
Wi-Fi connecting
Wi-Fi connected
CSV upload started
CSV upload completed
State: IDLE
```

## Этап 6. Задача со звездочкой: OTA-обновление

Реализовать режим обновления прошивки.

Условие входа в режим:

* длительное нажатие на кнопку.

Что нужно сделать:

1. По длинному нажатию перейти в OTA-сценарий.
2. Подключиться к Wi-Fi.
3. Перейти в состояние `OTA_CHECKING`.
4. Обратиться к REST endpoint сервера.
5. Получить информацию о доступной версии ПО.
6. Если доступна новая версия, скачать прошивку.
7. Перейти в состояние `OTA_UPDATING`.
8. Во время OTA-обновления светодиод должен гореть желтым постоянно.
9. Выполнить OTA-обновление.
10. Перезагрузить устройство.
11. После перезагрузки вывести новую версию ПО в консоль.

Пример:

```text
Firmware version: v1.0.0
OTA check started
Available firmware version: v1.0.1
OTA update started
OTA update completed
Restarting...
Firmware version: v1.0.1
```

Идеальный результат:

* OTA запускается только по длинному нажатию;
* устройство проверяет доступную версию ПО;
* если версия новая, устройство скачивает прошивку;
* во время обновления светодиод горит желтым постоянно;
* после обновления устройство перезагружается;
* после перезагрузки в консоли отображается новая версия;
* при ошибке или прерывания обновления по питанию, устройство не должно превращаться в "кирпич";
* ошибка OTA отображается красной индикацией и логируется в консоль.
